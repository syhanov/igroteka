# Design Note M00: Выбор источника деплоя GitHub Pages

## Статус

Принято.

## Контекст

Для публикации проекта необходимо выбрать источник деплоя GitHub Pages. Возможные варианты:

- публикация из ветки (`main` или `gh-pages`);
- публикация из папки `/` (root) или `/docs`;
- публикация с помощью GitHub Actions.

Проект представляет собой статический сайт, написанный на HTML, CSS и JavaScript без использования фреймворков и этапа сборки.

## Решение

В качестве источника деплоя выбрана ветка `main` с публикацией из корневой папки (`/`).

Такое решение позволяет публиковать сайт сразу после слияния изменений в основную ветку без дополнительных действий. Для разработки используются отдельные feature-ветки, которые после завершения работы объединяются с `main`.

Корневая папка подходит для данного проекта, поскольку сайт не требует сборки и не содержит большого количества технических файлов. Все необходимые для публикации файлы уже находятся в готовом виде.

Также выбран деплой непосредственно из ветки, поскольку проект не использует React, Vue или другие инструменты, создающие каталог `dist` или `build`.

## Рассмотренные альтернативы

### Папка `/docs`

Этот вариант был отклонён, поскольку он в основном используется, когда в корне репозитория находятся исходные файлы проекта, а публикуемая версия хранится отдельно. В данном проекте необходимости разделять исходный код и публикуемые файлы нет, поэтому использование `/docs` только усложнило бы структуру репозитория.

### GitHub Actions

GitHub Actions предоставляет гибкий процесс автоматической сборки и публикации сайта. Однако проект не требует этапа сборки, тестирования или генерации статических файлов. Использование GitHub Actions добавило бы лишнюю конфигурацию без получения практических преимуществ.

## Последствия

### Положительные

- простая настройка GitHub Pages;
- отсутствие дополнительных инструментов сборки;
- публикация происходит автоматически после обновления ветки `main`;
- структура проекта остаётся простой и понятной.

### Отрицательные

- при дальнейшем усложнении проекта может потребоваться переход на GitHub Actions или использование отдельной папки для публикации;
- корневая папка может стать менее удобной, если в проекте появится большое количество служебных файлов.
